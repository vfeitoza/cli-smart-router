# AGENTS.md

## Project

`smart-model-router` is a native CLIProxyAPI plugin that registers a configurable virtual model and routes matching requests to a built-in provider/model.

## Commands

```bash
go test ./...
make build-local
make build-all
gofmt -w cmd internal
CGO_ENABLED=1 go build -buildmode=c-shared -o smart-model-router.so ./cmd/plugin
```

Build artifacts generated by `go build -buildmode=c-shared` are reproducible and should not be committed by default:

```text
smart-model-router.so
smart-model-router.h
smart-model-router.dylib
smart-model-router.dll
```

## Architecture

Use the existing minimal DDD layout:

```text
cmd/plugin/
  main.go                  # C ABI entrypoint and RPC method dispatch
internal/domain/
  config.go                # Plugin config and normalization
  model.go                 # Candidate and routing decision types
  policy.go                # Deterministic policy/scoring logic
internal/application/
  router.go                # model.route use case
  registrar.go             # model.register use case
  usage.go                 # usage.handle use case
internal/infrastructure/
  cliproxy.go              # Minimal CLIProxyAPI JSON/ABI contracts used by this plugin
  rpc.go                   # CLIProxyAPI RPC envelope helpers
  yaml_config.go           # config_yaml parser
  state.go                 # In-memory config/state holders
  runtime_state.go         # Non-sensitive runtime cache, catalog, pricing, and usage state
```

## Development Rules

- Keep code comments and config field descriptions in English.
- Keep docs intended for plugin users in English unless explicitly requested otherwise.
- Keep the domain package free of C, HTTP callback, and CLIProxyAPI host implementation details.
- Do not add interfaces with one implementation unless they protect the domain/application layer from ABI or host-callback details.
- Prefer deterministic routing as the fallback path for every advanced feature.
- Do not log prompts, request bodies, credentials, API keys, auth records, or response bodies.
- `virtual_model` must remain configurable. Default is `router:auto`.
- `model.route` must return `Handled: false` for every model that is not the configured `virtual_model`.
- V1 uses local minimal CLIProxyAPI contract structs because this workspace has Go 1.22 and the local CLIProxyAPI SDK requires Go 1.26.
- Revisit direct SDK imports only when the project toolchain is Go 1.26 or newer.
- When adding a new feature, behavior, architectural decision, or resolving an important implementation doubt, create a new incremental ADR in `docs/adr/`.
- Save important new decisions, behavior changes, and implementation constraints to codemem using the codemem memory tools.

## CLIProxyAPI Contract Notes

- Use `TargetKind: "provider"` for the default path so CLIProxyAPI keeps auth selection, request execution, logging, and usage accounting.
- Declare `executor` only when `executor_fallback.enabled` is true; current fallback is non-streaming only.
- `model_provider` is for providing models to CLIProxyAPI, not for reading the full host catalog.
- Reading CLIProxyAPI's public model list is done through `GET /v1/models` via `host.http.do` when configured.
- `/v1/models` may not include provider/capability metadata, so configured candidate metadata remains authoritative.
